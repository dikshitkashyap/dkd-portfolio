IOCL Generic Cards — Designing the Fallback Without Making It the Default

Designing across 2 surfaces: Same fallback feature designed twice, once for truck owners on the app, once for ops agents on the portal. Increased card issuances by 60% without any major change to the primary flow.

ROLESole product designer
PLATFORMAndroid (BlackBuck app) + Ops web portal
OUTCOME60% increase in IOCL card issuances
01

Problems

23%

Dropped before portal entry — mobile number and KYC verification failures

17%

Dropped at IOCL API rejection — fuel KYC issues and IOCL-side rejections

Only 52% of interested users ever got a card. The rest hit a wall somewhere in the IOCL verification chain and had no path forward.

02

The solution

Generic cards solve this by bypassing CID creation entirely. Instead of issuing a card under the truck owner's identity, the card is created under BlackBuck's own CID. No mobile verification. No KYC check. No IOCL portal dependency. The card works immediately for fuel transactions.

How do you give users a shortcut without encouraging them to take it?

A generic card is strictly a fallback. It exists for users who've genuinely hit a wall — whose OTP didn't arrive, whose KYC failed, whose IOCL application got rejected. It is not meant to replace the standard flow, which gives the truck owner a card under their own CID, their own rewards account, and full control over their balance.

But if the generic card path is easy to find and frictionless to use, both truck owners and ops agents will start reaching for it by default — not because they're being dishonest, but because systems shape behaviour more than instructions do. Tell an agent to only use the fallback for genuine exceptions, then make the fallback button identical to the primary action. Watch what happens under load.

03

The design challenge

I was designing the same feature for two completely different users — a truck owner stuck in the middle of onboarding, and an ops agent processing dozens of applications a day. The right answer for one was wrong for the other.

04

For the truck owner: Relief at the right moment

The standard IOCL card creation flow starts with consent, moves through application details, and arrives at OTP verification. This is where 23% of users were dropping off — the OTP either didn't arrive or arrived for the wrong number.

The generic card option is not shown at the beginning of the flow, or even at the OTP screen when it first loads. It only appears after the 60-second countdown expires — as an orange banner at the bottom of the screen, below the keyboard. By that point the user has already waited, already tried, and is already frustrated. The fallback arrives as a contextually relevant solution, not a shortcut that competes with the primary flow.

DECISION

Timing: Make the fallback contextually relevant, not a shortcut

The difference between an option and a rescue matters: an option invites comparison and deliberation. A rescue arrives when you're already stuck and just needs to get you moving again.

05

For the ops portal — same fallback, different design logic

The ops portal is where agents review applications and create CIDs for truck owners. 83% of cases are routine. The other 17% hit exceptions — unclear documents, minor KYC mismatches, temporary IDs — where the agent needs a way to still move the application forward.

Generic card creation exists on the portal too. But an agent is not a frustrated truck owner who's been waiting 60 seconds for an OTP. An agent is someone processing high volumes under speed pressure, with incentives tied to throughput. If the generic card path is as easy as the primary flow, it stops being a fallback and becomes the default.

DECISION

Friction: Make the responsible action natural, the risky action intentional

The generic card feature exists on the portal but costs something to use — deliberate friction in the form of structured reasons, a mode switch, and a moment of declared accountability. Not all friction is bad friction. Friction becomes harmful only when it impedes the right action. When applied to sensitive workflows, friction becomes a guide.

06

What this project was really about

The generic card feature isn't technically complex. The IOCL API integration is straightforward. The UI surface area is small. What made this project interesting was the design question underneath — how do you build a responsible shortcut?

The answer was different for each user. For the truck owner, the shortcut earns its way in — it only appears after the primary path has genuinely failed. For the ops agent, the shortcut exists but costs something to use — deliberate friction in the form of structured reasons, a mode switch, and a moment of declared accountability.

The responsible action should feel natural, and the risky action should feel intentional. That balance — between usability and accountability — is what makes systems trustworthy at scale.
